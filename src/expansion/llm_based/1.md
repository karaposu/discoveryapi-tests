# `request_query_variants` — design

The Phase A LLM call that produces semantic query variants for a single run. Implemented as one LangChain call with structured output.

## Signature

```python
def request_query_variants(
    query_spec: BaseModel,                  # any Pydantic model (B2BQuerySpec, or future variants)
    paradigm_seeds: List[str],
    num_variants: int = 8,
    chat_model: Optional[BaseChatModel] = None,
) -> List[QueryVariant]
```

The function takes a **generic Pydantic model** as `query_spec`, not a fixed type. It does not import or reference `B2BQuerySpec` directly — it just serializes whatever model it receives to JSON and hands it to the LLM. This keeps the function reusable for any future *QuerySpec shape (e.g. a hypothetical product-discovery or recruiting spec); `B2BQuerySpec` is one concrete instance the runner happens to pass in.

## What the parameters mean

**`query_spec`** — a Pydantic model instance describing the run's query. The function's contract with it is exactly one operation: `query_spec.model_dump_json()`. Any model with `natural_query` plus whatever structured fields are relevant works. For Task 2 this is always `B2BQuerySpec`.

**`paradigm_seeds`** — a list of string identifiers telling the LLM what kinds of variant transformations to produce. These are categories, not enum values, kept as plain strings so the function isn't coupled to `ExpansionParadigm`. For B2B the seeds Task 2 uses are:

```python
[
    "verbatim",                # baseline; no broadening
    "title_broadening",        # synonyms for the role/title terms
    "industry_broadening",     # synonyms for the industry/domain
    "geography_broadening",    # synonyms for the location
    "combined_broadening",     # two or more of the above together
]
```

The LLM treats the list as a menu — it produces variants across the seed dimensions and tags each one with which seeds shaped it. The runner doesn't tell it "produce exactly two title_broadening variants"; the LLM picks the distribution itself, subject to `num_variants`.

**`num_variants`** — target count for the response (default 8). The LLM may return slightly fewer if seed coverage doesn't divide cleanly; the function does not pad.

**`chat_model`** — optional injection for tests. Defaults to the configured `ChatOpenAI`.

## Output

```python
class QueryVariant(BaseModel):
    variant_id: str               # e.g. "var_001_title_broadening"
    seeds: List[str]              # which seed names from paradigm_seeds shaped this variant
    query_body: str               # the Google query body — no site: prefix, no -keyword exclusions
    rationale: str                # one-sentence explanation
```

The function returns `List[QueryVariant]`. Each variant's `query_body` is consumed downstream by Python-only Phase B decoration (`source_lane` prefix + `noise_control` negatives). The function does not add or assume any heuristic decoration.

## Flow (LangChain structured output)

```python
class _QueryVariantsResponse(BaseModel):
    """LangChain return wrapper. Lifted out of QueryVariant so the LLM
       returns a single object with a list field, which structured_output
       handles cleanly."""
    variants: List[QueryVariant]


_VARIANTS_SYSTEM_PROMPT = """You produce semantic query variants...
(see desc.md § Phase A for the full prompt)"""

_VARIANTS_USER_PROMPT = """Generate {num_variants} query variants.

Query spec:
{query_spec_json}

Paradigm seeds (use these as a menu of transformation styles):
{paradigm_seeds_list}

Return a QueryVariantsResponse."""


def request_query_variants(query_spec, paradigm_seeds, num_variants=8, chat_model=None):
    llm = chat_model or create_openai_chat_model(model=DEFAULT_OPENAI_MODEL)
    structured = llm.with_structured_output(
        _QueryVariantsResponse, method="function_calling"
    )
    chain = (
        ChatPromptTemplate.from_messages([
            ("system", _VARIANTS_SYSTEM_PROMPT),
            ("user",   _VARIANTS_USER_PROMPT),
        ])
        | structured
    )
    response = chain.invoke({
        "num_variants":        num_variants,
        "query_spec_json":     query_spec.model_dump_json(indent=2),
        "paradigm_seeds_list": "\n".join(f"- {s}" for s in paradigm_seeds),
    })
    return response.variants
```

That's the whole function. Three things to call out:

1. **`method="function_calling"`** — same constraint as the existing judge call. OpenAI's strict response_format rejects schemas with defaulted/optional fields; function-calling mode doesn't.
2. **The wrapper class `_QueryVariantsResponse`** — LangChain's `with_structured_output` returns a single object. Wrapping the list lets the LLM return one structured response; the caller unwraps `.variants`. Underscore-prefixed because it's an internal implementation detail.
3. **No retry / no validation pass** — if the LLM returns fewer than `num_variants`, the function does not pad or re-call. The caller sees whatever it returned. Adding retries is out of scope; Task 2 is the baseline.

## What the function does NOT do

- Does not call SERP.
- Does not apply heuristic decoration (site:, -keyword) — that's Phase B.
- Does not iterate or refine variants based on downstream signal.
- Does not classify the query (no "what kind of query is this" inference) — it just renders variants from the spec it's given.
